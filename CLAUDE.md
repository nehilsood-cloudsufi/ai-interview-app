# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository structure

This repo currently holds one active POC, `liveAvatar/`, for an AI-driven vendor-interview experience ("Resonance") built on HeyGen's LiveAvatar **FULL Mode** SDK. HeyGen's Custom LLM feature points the avatar's LLM at our own backend (a "gateway"), so conversation logic (the Host agent) runs in our process, not HeyGen's — there is no separate agent worker process, it's all in-request/in-process inside the one FastAPI app.

It has its own `backend/` (Python/FastAPI, dependency-managed with `uv`) and `frontend/` (React 19 + Vite + TypeScript, `oxlint` for linting).

(A second POC, `livekit-app/`, built directly on the LiveKit Agents framework, was removed — see git history if you need to resurrect it.)

## `liveAvatar/backend/`

Package layout: `app/main.py` wires up the FastAPI app (CORS, routers, static/SPA mount) — no separate agent worker process, and no more startup-time Gemini provisioning (that lifespan step was removed with the old single-context/legacy-LLM flow). Routes live in `app/routers/` (`concurrency.py`, `interview.py`, `sessions.py`, `transcripts.py`, `llm_gateway.py`), each backed by `app/services/`:
- `liveavatar_client.py` — the LiveAvatar HTTP calls (create/delete context, create/delete per-interview LLM config + secret, create session token, stop session).
- `interview_state.py` — the in-memory interview registry (`create`/`get`/`get_by_token`/`remove`/`prune_older_than`): one `InterviewState` per interview holding the vendor profile, turns, current questionnaire node, follow-up count, scout findings, `manually_edited_fields` (profile fields the vendor hand-edited via the PATCH endpoint below, locked against the Host's LLM-reported `profile_updates` — manual always wins), and post-interview `pipeline_status`/`scorecard`/`recommendation`.
- `interview_config.py` — loads and validates the per-domain questionnaires (`data/questionnaires/{domain}.yaml`, one complete linear script per domain — `get_questionnaire(domain)` lru_cache'd per domain, `get_start_node_id(domain)`, `list_domains()` for the `(id, title)` pairs) and rubric (`data/rubric.yaml`) into typed `QuestionNode`/`RubricCategory` dataclasses.
- `host_agent.py` — the live interview: a deterministic state machine around one structured Gemini call per vendor utterance (`{"reply", "answer_complete", "profile_updates"}`). The LLM only phrases the reply, judges answer completeness, and reports vendor-profile details just stated; all state mutation (turn history, advancing the linear question script, follow-up accounting, merging `profile_updates` into the vendor profile) happens in code. This is also how onboarding works — there's no intake form, the profile is captured conversationally by the `intro`/`confirm_profile` questionnaire nodes. `handle_turn`/`stream_turn` take a `mode: "avatar"|"chat"` (default `"avatar"`, byte-identical to before); `mode="chat"` appends `settings.host_chat_mode_prompt` so the text-chat fallback treats terse typed answers as complete instead of pressing for elaboration. Any HTTP/parse failure soft-fails to a generic "say that again" reply with state left untouched. Clocked interviews (prod tier: `state.max_session_seconds` set, avatar mode only) are paced in code against the session clock (`first_turn_at`, stamped on the first turn): <120 s remaining suppresses follow-ups (prompt suffix + forced advancement), <60 s skips to a canned closing (`settings.host_timeup_reply`, no LLM call) — HeyGen's cap gets +60 s grace (`prod_session_grace_seconds` in `sessions.py`) so it never cuts mid-sentence.
- `scout_agent.py` — post-interview-only company research: one Gemini **native** REST call (not the OpenAI-compatible endpoint) with `google_search` grounding enabled. Deliberately never sees the interview or informs the Host's questions — it runs strictly after the interview ends, as a pipeline step, so the interview stays unbiased. Never raises; soft-fails to an empty findings list.
- `evaluator_agent.py` — `score_interview` runs ONCE at finalize-adjacent pipeline time: a single holistic pro-model pass over the whole transcript (plus Scout's findings) producing the rubric scorecard (per-answer live scoring was deliberately removed — the vendor must not watch their own scores mid-interview, and whole-transcript judgment beats per-answer judgment). Raises on failure; the pipeline soft-fails to `scorecard: null` and still reaches `ready`.
- `coordinator_agent.py` — pure deterministic threshold rule (no LLM, no I/O) over the Evaluator's scorecard: strong overall → recommend advancing, borderline overall with weak categories → recommend a clarification call, otherwise no recommendation. Does not draft anything itself.
- `pipeline.py` — the one and only post-interview orchestrator (agents never call each other directly). `enqueue()` schedules `run()` as an in-process `asyncio.create_task`, with an explicit comment marking the seam where a GCP Pub/Sub publish/subscriber would replace it in prod. `run()` sequences Scout → Evaluator → Coordinator, mutating `state.pipeline_status` through `interviewed → scouting → evaluating → ready`, or `failed` on any unexpected exception (never raises; best-effort re-saves the record with `pipeline_status: "failed"`). `state.pipeline_status` is flipped to `"ready"` only *after* the final `transcript_store.save` succeeds, so a save failure can't leave a transient "ready" visible to a poll.
- `session_state.py` — in-memory active-session counter. It only decrements on an explicit `/api/session/stop` call (user clicks stop, or the frontend's orphaned-session cleanup). When LiveAvatar's server ends a session on its own (e.g. `MAX_DURATION_REACHED`), the frontend's `SESSION_DISCONNECTED` handler resets local UI state but never calls `/api/session/stop` — so the counter drifts upward over repeated sessions and only resets on backend restart. Verified live; this is original behavior, not introduced by the `app/` restructure.
- `transcript_store.py` — persists finalized interview records. Uses **Google Cloud Storage** when `GCS_BUCKET` is set (blob `transcripts/{session_id}.json`), otherwise writes local JSON to `transcripts_local_dir` (dev fallback, gitignored). Sync GCS/file work is offloaded via `asyncio.to_thread`; `google.cloud.storage` is imported lazily inside the functions. No try/except here — failures propagate to the caller (the `transcripts` router returns 500; `pipeline.py` catches it itself).
- `summary_service.py` — generates the interview summary by calling Gemini's OpenAI-compatible chat endpoint through the shared `gemini_client` helper (no SDK), on the pro-tier model (`gemini_pro_model`). Deliberately raises on every failure path; the `transcripts` router is responsible for soft-failing so a summary failure never loses the transcript.
- `gemini_client.py` — the one shared Gemini chat POST used by every agent. The configured models are `-latest` aliases (`gemini-flash-latest` fast tier, `gemini-pro-latest` for finalize-time scoring/summary); on a model-not-found 4xx it retries once with the pinned `*_fallback` model from config, so an alias hot-swap can never take the app down.

Gateway-only sessions: `POST /api/session` requires both a live `interview_id` (created via `POST /api/interview`) and `PUBLIC_BASE_URL` to be configured — it 400s without an `interview_id` and 503s if `PUBLIC_BASE_URL` is unset, since HeyGen needs a public URL to call back into `/llm/{interview_id}/v1`. There is no more legacy/non-gateway session mode.

`POST /api/interview` optionally takes `{"domain", "tier", "passcode", "duration_minutes"}` in its body (400s on an unknown domain or tier; domain defaults to `settings.default_domain`, tier to `"dev"`; `tier="prod"` additionally requires `passcode` == `DEMO_PASSCODE` and accepts `duration_minutes` 1..`PROD_MAX_SESSION_SECONDS`/60, default 5, stored as `state.max_session_seconds` → the session's `max_session_duration` — see known constraints); `GET /api/domains` lists the available `(id, title)` pairs for a domain picker. In production an admin assigns the vendor's domain — the frontend's dropdown is a dev stand-in for that. Interview state carries `domain` and `tier`; finalized transcript records carry `domain`.

`POST /api/interview/{id}/chat` is the low-bandwidth text-chat fallback (used when the frontend's network-quality hook suggests switching off video): it drives the exact same `host_agent.handle_turn` as the avatar's `/llm/{id}/v1/chat/completions` gateway call, but same-origin and unauthenticated, since it never leaves our own backend — it passes `mode="chat"` so the Host's prompt adapts to terse typed answers.

`PATCH /api/interview/{id}/profile` is the vendor-initiated partial correction of the Host-captured profile: any subset of fields, each overwriting `state.vendor_profile` and locking into `manually_edited_fields`, plus a `role="system"` note turn appended to `state.turns` (only when a value actually changed) so the Host/Evaluator see the correction. Available any time up to finalize — it 409s once `state.pipeline_status` is set (i.e. after the interview has been handed to `app/services/pipeline.py`), since editing a record already in/through scoring would be silently lost. `TranscriptTurn.role` is `"interviewer" | "candidate" | "system"` to accommodate these notes.

The transcript flow: the `POST /api/transcript/finalize` route calls `summary_service.generate_summary(...)` inside a try/except (on failure → `summary=""`, `summary_ok=False`, logged as a warning, transcript still saved), then `transcript_store.save(...)` (on failure → 500, and any `pipeline_status` claim made for idempotency is released so a retry can proceed). When the request carries a live `interview_id`, finalize also hands the interview off to `app/services/pipeline.py` in the background — the scorecard/insights/recommendation are not in the finalize response; the frontend polls `GET /api/interview/{id}/state` for them. `GET /api/transcript/{session_id}` reads a saved record back (404 if absent).

`app/config.py` centralizes all env vars and constants (API keys, base URL, avatar id, prompts, `public_base_url`, `questionnaires_dir`/`default_domain` (env `QUESTIONNAIRES_DIR`/`DEFAULT_DOMAIN`) and `rubric_path`). Transcript/summary config also lives here: `gcs_bucket` (`GCS_BUCKET` env — GCS vs local backend switch), `transcripts_local_dir`, `gemini_base_url`/`gemini_native_base_url`, the four Gemini model settings (`gemini_model`/`gemini_pro_model` `-latest` aliases + their pinned `*_fallback` names, all env-overridable), and `interview_summary_prompt` (the Markdown summary system prompt). `app/models.py` has the Pydantic request/response models (incl. `TranscriptTurn`, `FinalizeTranscriptRequest`/`Response`, `CreateInterviewResponse`, `InterviewStateResponse`, `ChatRequest`/`Response`). One-off ops scripts (Gemini/context setup, account inspection, cleanup polling, a manual concurrency smoke test) live in `scripts/`, not part of the served app.

`tests/` holds a pytest suite with 1:1 coverage of every `app/` module (respx for HTTP mocking, hand-rolled in-memory GCS fakes in `tests/fakes.py`, shared fixtures in `tests/conftest.py`). pytest/coverage config is in `pyproject.toml` (`dev` dependency group, `asyncio_mode=auto`, branch coverage on `app`). `.github/workflows/ci.yml` runs this suite on push/PR.

Known constraints (see `docs/KT.md` for full rationale/troubleshooting):
- Avatar **tier** is per-interview, chosen by URL path (NOT an env-var deployment mode, and NOT the avatar/chat `mode` concept): `/` → `tier="dev"` (free sandbox avatar `SANDBOX_AVATAR_ID`, `is_sandbox: true`, ~1-min HeyGen cap), `/prod` → `tier="prod"` (`PROD_AVATAR_ID`, `is_sandbox: false`, 2 credits/min, capped at `PROD_MAX_SESSION_SECONDS` default 600). Prod-tier interview creation is gated by `DEMO_PASSCODE` (403 wrong, 503 if `PROD_AVATAR_ID`/`DEMO_PASSCODE` unset; session creation re-checks `PROD_AVATAR_ID`). Pairing the sandbox avatar with `is_sandbox: False` causes LiveKit to time out — that's why the sandbox avatar is pinned to the dev tier. `main`/`dev` branches are purely code-stability branches; both tiers exist in every deployment. Deploy recipe in `docs/KT.md` §6.
- Gateway mode needs a public URL: `PUBLIC_BASE_URL` must point somewhere HeyGen can reach (a dev tunnel locally, the Cloud Run URL in prod) or `POST /api/session` 503s. There's no more legacy fallback mode or hardcoded frontend context/LLM IDs — the context and per-interview Custom LLM are provisioned server-side in `sessions.py` for every session.
- Deployed as a **single** Cloud Run container: the multi-stage `Dockerfile` builds the frontend, then the FastAPI backend serves the compiled static files directly (avoids CORS). `.gcloudignore` must exclude `node_modules`/`.venv`/`.env` or Cloud Build fails on incompatible local binaries.
- The pipeline runs as an in-process asyncio background task after `/api/transcript/finalize` returns — under Cloud Run's default request-based CPU allocation this can stall between requests; see the Cloud Run CPU-throttling note in `docs/KT.md`.

Commands (run from `liveAvatar/backend/`):
```bash
uv sync                                                  # installs runtime + dev deps (pytest, respx, ...)
PUBLIC_BASE_URL=https://<tunnel> uv run uvicorn app.main:app --port 3001 --reload   # gateway needs a public callback URL (npx -y localtunnel --port 3001)
uv run pytest                                            # run the test suite (config in pyproject.toml)
uv run pytest --cov                                      # with coverage report
uv run python scripts/smoke_test_concurrency.py          # manual concurrency/session-lifecycle check (not pytest)
```

## `liveAvatar/frontend/`

`App.tsx` is composition-only. The entry point is `StartScreen`, which `POST`s `/api/interview` to mint an `interview_id` before anything else happens (there's no more intake form/landing page) and lets the vendor pick avatar or text-chat mode; it also fetches `GET /api/domains` to offer an interview-domain `<select>` — a dev stand-in for the admin-assigned domain in production, hidden if the fetch returns nothing. `App.tsx` then wires the hooks (`hooks/`) — `useLiveAvatarSession` (SDK session lifecycle, mic/camera, orphaned-session cleanup, and transcript capture via the SDK's `USER_TRANSCRIPTION`/`AVATAR_TRANSCRIPTION` events into a ref that survives cleanup), `useChatInterview` (drives the same Host agent over `POST /api/interview/{id}/chat`, for the one-way avatar→chat network-quality fallback), `useVendorProfile` (polls `GET /api/interview/{id}/state` every 5s for the whole interview, in both avatar and chat modes, while the session/chat is active; stops on 404 or repeated failures; exposes `saveProfile`, which PATCHes only the changed fields to `/api/interview/{id}/profile`), feeding an editable `ProfileCard` — heading "here's what I captured" during onboarding, "Your details" after — visible for the whole interview in both modes, but hidden in chat mode once the interview ends (finalize begins) since a post-finalize edit would just 409, `useNetworkQuality`, `useConcurrencyPoll`, `useSessionTimer`, and `useInterviewSummary` — into presentational `components/` (`StartScreen`, `ChatPanel`, `AvatarVideoPanel`, `LocalVideoPanel`, `SessionControls`, `TranscriptPanel`, `SummaryPanel`, `FollowupPanel`, `ProfileCard`, etc.). When a session ends (user-stopped or server-ended), `useLiveAvatarSession`'s `onSessionEnd` hands the captured turns to `useInterviewSummary.finalize`, which POSTs to `/api/transcript/finalize` and then, in gateway mode, polls `GET /api/interview/{id}/state` every 3s until the pipeline reaches a terminal status — progressively filling in `SummaryPanel`'s scorecard/insights/follow-up as they arrive rather than waiting for everything at once. `utils/downloadTranscript.ts` builds a downloadable Markdown record. `config.ts` holds `API_URL`, the runtime `TIER` constant (`'prod'` when `location.pathname` starts with `/prod`, else `'dev'` — StartScreen shows a passcode field + "Production" badge on the prod tier and sends `{tier, passcode}` with `POST /api/interview`), plus two build-time Vite flags: `VITE_SHOW_SELF_VIEW` (default on; hides the local camera panel and skips camera permission when set to `false`) and `VITE_SESSIONS_SHEET_URL` (optional link to an "all sessions" sheet, rendered only when set). The backend's 404→`index.html` middleware makes `/prod` load the SPA on Cloud Run.

Commands (run from `liveAvatar/frontend/`): `npm install`, `npm run dev`, `npm run build`, `npm run lint`.

Required env vars: `LIVEAVATAR_API_KEY`, `GEMINI_API_KEY`, and `PUBLIC_BASE_URL` (all in `backend/.env.example`; gateway mode is the only mode and session creation 503s without a `PUBLIC_BASE_URL` HeyGen can reach). Optional: `GCS_BUCKET` (GCS transcript backend; local JSON when unset), `SCOUT_ENABLED` (default true), `HOST_STREAMING_ENABLED`, and the prod-tier trio `PROD_AVATAR_ID`/`DEMO_PASSCODE`/`PROD_VOICE_ID` + `PROD_MAX_SESSION_SECONDS` (see known constraints above). Frontend build-time flags live in `frontend/.env.example`.

Deployment: `deploy_setup.sh` provisions the `LIVEAVATAR_API_KEY` secret in Google Cloud Secret Manager and binds IAM so Cloud Run can read it (reads the key from `$LIVEAVATAR_API_KEY`, does not hardcode it) — run this before `gcloud run deploy`. `.github/workflows/ci.yml` runs frontend lint/build and, for the backend, an import-sanity check plus the full pytest suite on push/PR.
