Summary
The team finalized chat-based onboarding workflows, decoupled agent architectures, and sequential processing strategies for the interview platform.

Interview Flow and Onboarding
Onboarding processes integrate directly into chat functionality to gather candidate details without document uploads. This approach utilizes a sequence of 20 domain-specific questions to ensure consistent information collection.

Technical Architecture and Execution
The system utilizes Gemini 3.5 Flash for transcription and Gemini 3.1 Pro for evaluation via a decoupled, sequential pipeline. Task queuing optimizes performance by enabling immediate processing following each interview session.

Agent Responsibilities and UX
Modular design principles separate internet data scouting from evaluation logic to maintain system flexibility. A decision was made to retain participant video visibility for psychological responsiveness while enabling toggles for configuration.




Decisions
Aligned
* Document upload functionality removal The document upload capability is removed from the chat interface, moving to direct data collection through chat.
* Interview process question structure The interview process is structured to collect four static onboarding questions followed by a predefined set of twenty interview questions.
* Low bandwidth text fallback implementation A text-based chat fallback mechanism is implemented for low bandwidth situations.
* Asynchronous Data Scout Agent execution The Data Scout Agent is decoupled from the live interview flow to run as an asynchronous process.
* System workflow agent sequence The system workflow is established as a sequential process: Interview, followed by Data Scout Agent, then Evaluator Agent, and finally Coordinator Agent.
* LLM model selection policy The LLM strategy is defined as using Gemini 3.5 Flash for real-time interview hosting and Gemini 3.1 Pro for final evaluation and summary generation.
* NoSQL JSON data storage Session data, transcripts, and summaries are stored in a NoSQL database using JSON format.
* Optional user video feed The user video feed display is made optional and controlled by a feature flag.
* Landing page removal The landing page is removed from the application flow.

* Onboarding Setup and Chat Functionality: Nehil Sood proposed a simple onboarding setup that includes capturing details such as company name, website, contact name, and role for a candidate appearing for an interview, along with an option to upload company documents that are not available on the internet. Aditya Sai Srikanth Palaparthy and Suryarjuna Gopakumar clarified that the client requested this process to be completed entirely through chat, noting that the client explicitly rejected having an upload functionality (00:01:11). Consequently, the team reached a consensus to remove the document upload capability and the landing page entirely, opting to collect the person's name, role, company, and website link directly through the initial chat questions (00:02:16) (00:21:22).
  

* Integration of Onboarding into the Live Interview: Aditya Sai Srikanth Palaparthy explained that onboarding will not be a standalone step but will be combined with the interview process itself. The live agent will ask an initial set of questions to capture the company name, email ID, or website, and will then display a list to the candidate to confirm if the captured details are correct before proceeding with the interview (00:02:16). Suryarjuna Gopakumar summarized the structure, stating that the first four questions of the interview will be static information-gathering questions, which will be followed by a predefined domain-specific set of 20 questions (00:03:13).
* Natural Conversation Flow and Text-Based Fallback: Nehil Sood raised concerns that asking rigid questions upfront could break the flow of the interview, suggesting instead that the AI gather information naturally (e.g., from an open introduction like "Hi, I'm Nehil Sood from Cloud Sufi...") and then segregate it into a structured response for the agents (00:04:12). Suryarjuna Gopakumar agreed with this approach to ensure a natural flow (00:05:17). Additionally, Suryarjuna Gopakumar highlighted the need for a text-based fallback method for situations with low bandwidth where video is not supported. In this fallback mode, the system will switch from the live avatar to a text screen that asks questions or infers and confirms previously stated information, such as checking if a vendor supports clouds other than GCP (00:06:24).
  

* Decoupled Architecture and Sequential Agent Processing: The team discussed the timing and execution of the data scout agent. Nehil Sood noted that KI had a different understanding, believing the agent would run asynchronously in the background and tweak the interview in real time (00:09:09) (00:14:35). Suryarjuna Gopakumar argued against this approach, explaining that since the interview strictly follows the predefined 20 questions regardless of the vendor's inputs, running the agent in real time is unnecessary and risks introducing bias (00:08:12) (00:10:10). Suryarjuna Gopakumar emphasized the design principle of building separate "Lego blocks" to keep the system lightweight and unbiased, noting that the blocks can easily be merged in the future if needed. The team agreed on a sequential process where the interview is completed first, followed by the data scout agent gathering internet data, then the evaluator agent assessing the candidate, and finally the coordinator agent aligning with a human evaluator (00:10:10) (00:12:43).
  

* API Usage and Data Storage Strategy: Nehil Sood outlined the technical execution for the interview API calls, stating that the system will utilize Gemini 3.5 Flash for the live interview transcription and host management due to its efficiency. Once the interview is complete, the entire transcript will be sent to Gemini 3.1 Pro via a single API call for the heavier, more intelligent tasks of evaluation and summary generation (00:14:35). Nehil Sood and Suryarjuna Gopakumar also agreed on the data storage strategy, concluding that transcripts, session summaries, and overall session details must be stored in a proper NoSQL JSON format so that a human evaluator can easily review the session later (00:17:29).
* Asynchronous Pipeline and Execution Timing: To optimize performance and minimize usage during Peak Application Times (PAS), Suryarjuna Gopakumar suggested exploring bulk processing, such as running data validations and evaluations together every four hours or after a 30-minute delay (00:15:49). Nehil Sood recommended firing the process immediately after an interview by queuing the tasks in a message queue (using a pub/sub methodology), arguing that it only requires two or three sequential Gemini calls and will notify users as soon as the evaluation and summary for a vendor are ready. Suryarjuna Gopakumar agreed to this approach, noting that the user interface will simply need a table to track statuses, such as "pending processing" or "pending evaluation," to keep users informed (00:16:36).
  

* User Video Visibility and Stakeholder Context: Nehil Sood noted a comment from the client requesting the removal of the user's live video from the top right of the screen, leaving only the AI assistant visible. Suryarjuna Gopakumar critiqued this design, explaining that psychological principles show people are more polite, honest, and responsive when they can see themselves, comparing it to the mirrors placed behind reception desks (00:18:32). Nehil Sood proposed a technical compromise to keep a feature flag or hook in the code so the video can be toggled off easily if the client insists. Additionally, Suryarjuna Gopakumar clarified the roles of the stakeholders involved: Lana is a TPM/mediator from SAIL (a business unit within Aramco), Caddy is a project manager from their end, and G (or Gokati) is from Aramco.ai, which operates as a separate business unit with independent management (00:19:29).
* Data Scout Agent Responsibilities and Modular Design: Karan Tripathi questioned whether the data scout agent should flag discrepancies or findings within the interview answers. Suryarjuna Gopakumar clarified that the data scout agent's sole responsibility is to independently collect information about a company from the internet and store it, thereafter alerting the evaluator agent that the vendor is ready for processing. Evaluating the interview transcript against the internet findings using a scoring rubric is strictly the evaluator agent's job (00:22:23). Suryarjuna Gopakumar reiterated that maintaining this separation allows the data scout agent to remain modular, enabling it to be repurposed for other tasks in the future, such as acting as a LinkedIn scraper for hiring (00:23:25) (00:25:44).
* Handover Planning and Wrap-Up: Nehil Sood informed the team that they will be on leave starting December 24th. To ensure project continuity, Nehil Sood confirmed they are keeping Karan Tripathi in the loop and will pass on their current work (00:25:44). Nehil Sood mentioned they are currently managing a separate repository and will share a proper working version of the code once it is ready (00:26:39). Suryarjuna Gopakumar agreed to take over the code and move it forward from there (00:25:44). The meeting concluded with Aditya Sai Srikanth Palaparthy noting that their limited availability was due to managing three concurrent tasks, including an ongoing UAT (00:26:39).
